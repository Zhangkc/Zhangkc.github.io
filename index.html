使用perf对nodejs应用进行性能测试

准备工作

  • 设置最大文件句柄
  • linux下tcp参数设置
  • 保证压力源和被测服务器在同一个网段（防止nat转换端口用尽）
  • 服务器监控程序，如果没有可以自行安装netdata等
  • 保证压力源和被测服务器之间带宽符合要求

常用指令

采集nodejs应用的火焰图

启动nodejs应用时增加参数--interpreted-frames-native-stack(Node.js 10+)，--perf-basic-prof-only-functions 或 --perf-basic-prof
如果没有这些参数的话，你照样可以得到火焰图。不过大部分的横条里的函数调用都将被标记为 v8::Function::Call

之后使用perf命令采样即可，获取数据后借用FlameGraph开源库里面的stackcollapse-perf.pl和flamegraph.pl生成火焰图

当然也可以使用--inspect参数启动nodejs应用，再通过chrome的开发者工具查看火焰图。

看懂火焰图

刚刚我们生成了nodejs的火焰图，可以发现其中有几种不同的颜色，含义如下：
| 颜色 | 类型 |
| — | — |
| 绿色 | 应用js代码 |
| 蓝色 | nodejs库代码 |
| 黄色 | 引擎C++/C代码 |
| 红色 | 系统调用 |

Y轴表示栈的深度，X轴表示总样例，左右顺序按照首字母排列，无特殊意义

常见帧栈prefix的说明：
|帧栈prefix|含义|
|—|—|
|LazyCompile| V8引擎开发者原话: LazyCompile prefix does not mean that this time was spent in compiler, it just means that the function itself was compiled lazily. 表示该函数是延迟编译的|
|Builtin|内置的方法|
|Stub|C入口桩代码：作用是在js的JIT代码中，如果要调用Runtime的函数，则通过CEntryStub实现|
|V8::internal V8::Function|V8引擎内部C++的命名空间和函数调用|
|*(asterisk)|表示该函数被V8引擎优化过|
|~(tilda)|表示该函数未被V8引擎优化|

Sec-WebSocket-Key是一个16字节的随机值，并经过base64编码

Sec-WebSocket-Protocol表示客户端可使用的子协议，由服务器选择一个子协议与客户端通信，子协议是由websocket承载的协议。

Sec-WebSocket-Version必须，并且草案里规定值为13

Sec-WebSocket-Accept必须，服务端根据客户端发送的Sec-WebSocket-Key和258EAFA5- E914-47DA-95CA-C5AB0DC85B11进行拼接，并经过sha-1加密后再用base64编码，返回给客户端验证。

客户端验证成功后握手就完成了。

数据帧

草案里定义的帧结构图

Fin: 1bit

表示该帧是否为整个消息的最后一帧。

RSV1, RSV2, RSV3: 1bit each

保留位，一般都是0，可以用于websocket扩展协议的定义。

Opcode: 4bits

表示数据包的类型
|值|定义|
|:-|:-|
|%x0|表示是上一个帧的延续|
|%x1|表示是一个字符帧|
|%x2|表示是一个字节帧|
|%x3-7|保留值，供未来的非控制帧使用|
|%x8|表示关闭连接|
|%x9|表示心跳请求ping|
|%xA|表示心跳响应pong|
|%xB-F|保留值，供未来的控制帧使用|

Mask: 1bit

表示数据包是否经过掩码转换。为了中间缓存攻击，从客户端发往服务端的数据必须通过一个mask key进行异或运算

假设payload长度为pLen，mask key长度为mLen，转换方法如下：

Payload: 7 bits, 7+16 bits, or 7+64 bits

  1. 读取9-15位 (包括9和15位本身)，并转换为无符号整数。如果值小于或等于125，这个值就是长度；如果是 126，请转到步骤 2。如果它是 127，请转到步骤 3。
  2. 读取接下来的 16 位并转换为无符号整数，并作为长度。
  3. 读取接下来的 64 位并转换为无符号整数，并作为长度。

Masking-key: 0 or 4 bytes

Mask为1时长度为4字节，随机生成。

关闭连接握手

关闭连接的时候，只用发送一个opcode为0x08的帧，payload中前2个字节写入定义的code，后续写入关闭连接的reason，那么一个关闭流程就握手就开始。如果对端没有响应，在一段时间后直接关闭tcp连接。

其他

通过抓包还发现了Ethernet Frame会自动padding，即在报文末尾补充0。实际上这是由于IEEE 802.3规定以太网帧最小长度是64字节，如果小于64字节就会最后补0。但是我们在wireshark里看到的最小的以太网帧始终都显示60字节，这又是为什么呢？其实wireshark会自动省略掉帧中的最后4个字节，即FCS(Frame Check Sequence)，它是用来检测帧是否损坏的。它的值是对帧数据进行某种运算（冗余循环校验，可以理解为对数据的一种逻辑摘要）的结果，接收方在收到FCS后，对帧数据进行相同的运算，结果与收到的FCS对比从而检查数据是否出错。

根据费茨法则(Fitts’Law)，使用指点设备达到一个目标的时间与一下两个因素有关：

  1. 起始位置和目标中心的距离。距离越长，所用时间越长
  2. 目标的大小。目标越大，所用时间越短

因此扩大目标元素大小是提高UI体验很好的方法。

跨域请求

所有跨域提交某些请求的时候，如果服务端设置了可跨域访问，都会默认发送两次请求

  • 第一次是预检请求，来查询是否支持跨域
  • 第二次才是真正的请求

那有没有办法只让js发送一次请求呢？其实js中出现这个现象原因在于你发送了一个非简单的请求，所谓非简单请求：

只使用GET HEAD 或者POST请求方法，

http头的信息不超出一下几种字段

  • Accept (发送方希望收到的数据类型)
  • Accept-Language
  • Content-Language
  • Last-Event-ID
  • Content-Type

如果使用POST向服务器传送数据，则数据类型(Content-Type)只能是

  • application/x-www-form-urlencoded
  • multipart/form-data
  • text/plain

对于简单请求，浏览器直接发出CORS请求，就是在通用头部增加一个Origin字段。
非简单请求的CORS请求，会在正式通信之前，增加一次HTTP查询请求，称为”预检”请求(preflight)。

AMF格式分析

AMF

是Action Message Format的简写，Adobe推出的一个二进制数据格式。其优点在于数据压缩率高，可以发送复杂数据，比如自引用的数据等。由于最近需要在web上收发这个东西，并且js没有现成解析AMF的库，网上的大部分都是只能解析简单数据，就仔细（并不）研究了一下。

解析

AMF分为AMF0和AMF3，AMF3是AMF0的下一个版本，这里我们主要说一下AMF3的解析

文档

AMF 0 ：http://wwwimages.adobe.com/content/dam/Adobe/en/devnet/amf/pdf/amf0-file-format-specification.pdf

AMF 3 ：http://download.macromedia.com/pub/labs/amf/amf3_spec_121207.pdf

首先明确一个uint29的概念，AMF3采用一种特殊的格式来表示整数。对于一个正常的整数，一般用4个字节储存，但是AMF为了便于压缩，整数的储存是变长的。AMF最少用1个字节来表示一个整数，最多用4个字节。对于4个字节的整数，前3个字节的最高位都是用来标识下一个字节是不是该整数的一部分。这意味着32个字节中有3个字节是标识位，只用剩下的29位来表示整数，因此最大的无符号整数值为2^29-1。

我们直接来看数据

  • 00 03 表示版本号为3
  • 00 00 表示报文头部个数为0。如果不为0，后面紧跟具体的头部内容。每个头部内容的格式为

    小节名称 + 16位BE编码长度 + UTF-8编码字符串内容 + 8位布尔值(must-understand 标记) + 32位BE编码小节长度 + AMF0值

  • 00 01 报文体个数，0001表示有一个报文体
